# SPEC

## 1. 任务目标
完成项目第一阶段：构建一个可运行、可验证、可调试的掼蛋基础系统。

该基础系统由两部分组成：

1. **掼蛋引擎**
2. **基础掼蛋 AI**

第一阶段的最终目标不是“最小规则子集可跑”，而是形成一个能够支撑后续第二阶段自主学习的**稳定基础环境**。

因此，第一阶段必须满足以下总要求：

- 4 个 AI 可以按照当前第一阶段规则完成完整自动对局
- 系统在已支持规则范围内能够正确出牌、正确压牌
- 非法动作不会进入状态更新
- 开发者能够通过全信息调试确认每一步动作是否正确
- 基础 AI 不只是合法，而且在已支持规则范围内应表现出基础人类经验
- RAG 与后续 DeepSeek API 接入只能作为知识支持链路，不能替代引擎真值

当前项目已经在部分规则子集上实现了正确出牌与正确压牌，因此第一阶段后续工作的重点不是重写基础骨架，而是：

1. 补全基础掼蛋所需的规则覆盖
2. 补全规则库与经验库正文
3. 接入 DeepSeek API
4. 在更完整规则范围下重新验证 4 AI 自动对局稳定性

## 2. 第一阶段的核心验证顺序

### 2.1 第一优先：4 AI 自动对局闭环
第一阶段首先必须支持 4 个 AI 从初始发牌开始，完成一局完整自动对局，用于验证：

- 掼蛋引擎是否闭环
- 状态转移是否正确
- 合法动作生成是否正确
- AI 是否始终只从合法动作集合中选择动作
- AI 是否存在规则误解
- 系统是否存在死循环、非法动作、状态污染

### 2.2 第二优先：开发者全信息观察
在 4 AI 自动对局测试阶段，开发者必须能够看到：

- 4 个 AI 的全部手牌
- 当前轮到哪一个 AI
- 当前桌面动作或当前轮约束
- 当前 AI 的合法动作集合
- 当前 AI 的最终选择
- 动作执行前后的关键状态变化
- 当前局面推进情况
- 最终对局结果

这样才能判断：

- 每个 AI 是否都在正确出牌
- AI 是否只是“没报错”，还是确实按规则、按经验在出牌
- 某一步异常动作到底是规则错误、策略错误还是知识理解错误

### 2.3 第三优先：后续接入人类玩家
人类玩家接入不是第一阶段最前置的核心验收项。

第一阶段的首要目标是先通过 4 AI 自动对局和全信息调试，确认：

- 规则引擎正确
- 基础 AI 行为基本合理
- 系统稳定

只有在此基础上，才适合进一步引入人类玩家。

## 4. 第一阶段规则范围

### 4.1 规则版本约定
第一阶段必须采用**项目内部冻结的一套“完整基础掼蛋规则”版本**。

这里的“完整基础掼蛋规则”不是指所有比赛扩展规则都必须完成，而是指：

- 足以支撑“基础掼蛋 AI”成立
- 足以让普通人认为这确实是在打掼蛋
- 足以为第二阶段自主学习提供稳定环境

规则文本、知识库正文与代码实现必须始终保持一致。

### 4.2 第一阶段必须支持的流程规则
第一阶段至少必须支持以下流程规则：

1. **发牌**
   - 支持初始化 4 名玩家手牌
   - 支持从开局进入首个行动玩家状态

2. **轮流出牌**
   - 任一时刻只能有一个当前行动玩家
   - 玩家行动顺序固定循环推进

3. **pass**
   - 当当前玩家在跟牌阶段不选择出牌时，可执行 pass
   - 主动领出阶段不允许以 pass 代替出牌
   - pass 不改变玩家手牌

4. **一轮结束**
   - 当从领出玩家开始，其他玩家均连续 pass，且无人再压制当前桌面动作时，本轮结束
   - 一轮结束后，桌面约束清空
   - 上一轮最后一个成功出牌的玩家获得下一轮领出权

5. **一局结束**
   - 当任一玩家手牌数量为 0 时，该玩家视为出完牌
   - 第一阶段最小要求是能正确判定终局并结束当前验证局
   - 更复杂的名次与比赛制结算可暂不作为第一阶段核心要求

6. **4 AI 自动对局闭环**
   - 从初始化到终局必须可完整推进
   - 不得出现无法推进、死循环或状态污染

## 5. 第一阶段牌型范围

### 5.1 当前已实现的基础子集
当前项目已实现的基础子集包括：

- 单张
- 对子
- 三张
- 炸弹（当前最小定义）
- pass

### 5.2 第一阶段必须补齐的规则范围
第一阶段不应停留在上述最小子集，而必须继续补齐以下内容：

- 大王、小王相关表示
- 大王、小王相关比较规则
- 大王、小王参与的动作生成
- 你认定“基础掼蛋”不可缺少的主要牌型
- 与这些牌型对应的合法动作生成
- 与这些牌型对应的压牌规则
- 与这些牌型对应的状态转移与回归测试

### 5.3 当前最小已冻结牌型定义
在当前已实现子集内，最小定义仍为：

1. **单张**
   - 由 1 张牌组成

2. **对子**
   - 由 2 张点数相同的牌组成

3. **三张**
   - 由 3 张点数相同的牌组成

4. **炸弹**
   - 当前最小定义为：4 张点数相同的牌
   - 当前不应把更复杂炸弹体系误写成“已实现”

5. **pass**
   - 不是牌型，而是合法动作类型
   - 仅在跟牌阶段允许

### 5.4 后续补齐规则的写法要求
在规则库正文和代码实现中，必须明确区分：

- **当前已实现规则**
- **第一阶段待补规则**
- **仅作背景说明的完整规则**

未实现规则不得被写成“当前系统已支持并可执行”的规则。

## 6. 当前最小比较规则与后续补齐原则

### 6.1 当前已实现比较规则
在当前已实现子集内，比较规则仍为：

#### 同型比较
- 单张与单张比较：点数更大的动作更大
- 对子与对子比较：点数更大的动作更大
- 三张与三张比较：点数更大的动作更大
- 炸弹与炸弹比较：点数更大的动作更大

#### 异型比较
- 当前普通牌型之间不得跨型比较
- 当前唯一例外：
  - 炸弹可以压制单张、对子、三张
  - 炸弹之间按炸弹点数比较

#### 跟牌规则
- 若当前桌面约束为空，则当前玩家必须从合法动作集合中选择一个非 pass 动作领出
- 若当前桌面约束非空，则当前玩家只能：
  1. 出一个能够合法压制当前桌面动作的动作；或
  2. 选择 pass
- 当前桌面为单张/对子/三张时：只允许同型更大，或炸弹
- 当前桌面为炸弹时：只允许更大炸弹，或 pass

### 6.2 后续补齐原则
当第一阶段继续补齐更完整牌型时，必须同时补齐：

- 牌型识别
- 合法动作生成
- 跟牌与压牌规则
- 状态转移
- 测试
- 规则知识库正文
- 经验知识库中对应的经验条目

## 8. 基础掼蛋 AI 要求

### 8.1 基础 AI 不等于随机 agent
基础 AI 不能只是：

- 随机从合法动作中取一个动作
- 只要不报错就算完成
- 没有任何经验偏好

### 8.2 基础 AI 的最低要求
基础 AI 至少应当是：

- 规则约束下的非学习型 baseline
- 具备基础人类经验先验
- 能够完成完整自动对局
- 行为上接近普通人的基础打法

### 8.3 当前基础 AI 的定位
当前基础 AI 已具备：

- 合法动作集合内选择能力
- 基础规则型与经验型启发式
- 基础可解释性

但在第一阶段收口前，还需要在更完整规则范围下继续验证其稳定性与合理性。

### 8.4 “像正常人出牌”的最低要求
第一阶段不要求 AI 很强，但必须满足以下最低行为标准：

1. 不能是纯随机合法动作采样器
2. 不能在同一固定局面中频繁给出无明显依据的完全不同动作
3. 不能无意义优先打出明显高价值控制牌
4. 不能无意义拆解明显应保留的基础组合
5. 在固定测试局面中，应优先选择至少一个明显更合理的基础动作
6. 其行为应能给出基础规则依据或经验依据，而不是黑箱随机输出

## 9. RAG 与知识库定位

### 9.1 RAG 的职责
RAG 在第一阶段中的职责是：

1. 为 AI 提供掼蛋规则知识检索
2. 为 AI 提供人类经验知识检索
3. 降低模型误解规则的风险
4. 降低模型引用其他纸牌游戏规则的风险
5. 帮助 AI 在合法动作集合内做出更像正常人的选择

### 9.2 RAG 不承担的职责
RAG 不应承担以下职责：

- 不应替代规则引擎
- 不应作为动作合法性的最终裁判
- 不应绕过合法动作生成器直接输出可执行动作
- 不应作为状态转移真值来源
- 不应成为掼蛋规则程序实现的替代品

### 9.3 规则真值来源
动作是否合法、状态能否更新，最终必须由**程序化规则引擎**决定，而不是由 RAG 检索结果或模型输出决定。

### 9.4 知识库要求
规则知识库与经验知识库必须：

- 与当前实现一致
- 与当前冻结规则范围一致
- 未实现规则必须显式标注为“未接入执行链路”
- 不得把背景规则误写成当前可执行规则

---

## 10. DeepSeek API 定位

### 10.1 第一阶段中的作用
DeepSeek API 在第一阶段中的作用只能是：

- 为规则解释提供模型支持
- 为经验检索结果提供模型支持
- 为后续更强的知识辅助链路做准备

### 10.2 不允许的用途
DeepSeek API 不得：

- 替代引擎做动作合法性判断
- 替代引擎做状态转移裁决
- 绕过 `legal_actions` 直接生成并执行动作
- 把模型输出当成规则真值

### 10.3 接入要求
DeepSeek API 接入必须满足：

- 配置通过 `.env` 和 `config.py` 读取
- 不得写死 API key、base URL 或模型名
- 失效或不可用时不得破坏规则真值链路

## 11. 调试输出要求

### 11.1 内部日志要求
系统必须保留结构化调试日志，用于：

- 测试断言
- 回归验证
- 问题复盘
- 机器可读的调试记录

### 11.2 默认终端输出要求
`run_4ai_debug.py` 的默认终端调试输出必须采用**人工可读格式**，便于开发者手工检查“是否正确出牌、是否正确压牌”。

1. 发牌后 4 名玩家手牌

3. 当前轮结束后的手牌概览
