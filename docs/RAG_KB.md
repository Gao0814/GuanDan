# RAG_KB

## 1. 目标
第一阶段中的 RAG 只服务于两个核心目标：

1. 限定 AI 对掼蛋规则的理解范围
2. 引入基础人类经验，使 AI 更像正常人出牌

在后续接入 DeepSeek API 后，RAG 还承担第三个目标：

3. 为模型提供受限的规则与经验上下文，但不改变规则真值链路

---

## 2. 主知识源约定

### 2.1 规则知识库主文件
第一阶段的**唯一主规则正文**为：

- `rag/rule_corpus/guandan_rules.md`

该文件是以下内容的主文本依据：

- 当前已实现规则的文字定义
- 第一阶段待补规则的文字定义
- 背景规则与完整规则说明

除非你主动修改这份文件，否则项目中不应再维护第二份具有同等地位的规则正文文件。

### 2.2 经验知识库主文件
第一阶段的**唯一主人类经验正文**为：

- `rag/experience_corpus/basic_human_experience.md`

该文件是以下内容的主文本依据：

- 当前基础 AI 已采用的经验规则
- 可用于解释当前 AI 选择逻辑的经验条目
- 第一阶段待接入或待强化的经验条目

### 2.3 文档与代码的从属关系
在第一阶段中，应遵循以下关系：

- `guandan_rules.md`：规则正文主文件
- `basic_human_experience.md`：经验正文主文件
- `docs/SPEC.md`：项目目标、边界、当前状态
- `docs/TESTS.md`：验证规则与经验是否正确落地
- 代码实现：以规则引擎为真值执行层
- RAG：以知识库正文为检索层
- DeepSeek API：以后续受限模型支持层的身份接入

也就是说：

- 规则正文不应再由其他说明文档“重复定义一份”
- 其他文档必须与这两份正文保持一致
- 代码实现、测试、RAG 检索必须围绕这两份正文收敛

---

## 3. 知识库分层要求

### 3.1 规则知识库分层
`guandan_rules.md` 应显式区分以下三层内容：

#### A. 当前已实现规则
指当前代码已经支持、当前测试已经验证、当前可以进入执行链路的规则。

这部分内容可以被：

- 规则检索
- 调试解释
- DeepSeek 上下文引用

直接用于当前局面的规则支持。

#### B. 第一阶段待补规则
指你已经确定要纳入第一阶段，但当前代码尚未完成实现的规则，例如：

- 大王、小王相关处理
- 当前尚未补齐的基础掼蛋牌型
- 与这些牌型对应的压牌规则
- 与这些牌型对应的动作生成与状态转移规则

这部分内容可以写入知识库正文，但必须显式标注：

- 当前未接入执行链路
- 当前不能作为规则真值
- 当前不能直接用于动作合法性判断

#### C. 背景规则或完整规则说明
指帮助理解完整掼蛋的背景规则、扩展规则或未来可能用到的规则。

这部分只能用于：

- 背景说明
- 人工阅读
- 未来扩展参考

不得被误用为当前可执行规则。

### 3.2 经验知识库分层
`basic_human_experience.md` 应显式区分以下两层：

#### A. 当前已用于 AI 的经验
这部分经验应与当前基础 AI 的启发式保持一致，例如：

- 记录高价值信息
- 尽量减少未来出牌轮次
- 不无意义提前消耗炸弹
- 不无意义拆解基础组合
- 队友少牌时优先配合
- 读人与反推只作为弱信号，不替代规则真值

这些经验可直接用于：

- 合法动作集合内排序
- 决策依据输出
- 调试解释
- DeepSeek 上下文支持

#### B. 当前已收集但尚未接入排序逻辑的经验
这部分经验可以存在于知识库中，但必须承认其状态：

- 当前尚未进入执行排序
- 当前只是候选经验
- 当前不应在代码实现中被假定为已生效

## 4. 来源约束

### 4.1 允许的知识来源
第一阶段中，RAG 默认只允许从以下两份文件中读取正文知识：

- `rag/rule_corpus/guandan_rules.md`
- `rag/experience_corpus/basic_human_experience.md`

如无额外文档更新，不应再加入第三份具有同等地位的规则正文或经验正文。

### 4.2 内容质量要求
规则知识与经验知识必须满足：

- 与当前项目采用的规则口径一致
- 与 `docs/SPEC.md` 中声明的当前状态一致
- 不包含其他纸牌游戏规则
- 经验条目不与规则冲突
- 不引入模糊、不确定、互相矛盾的知识

### 4.3 当前经验知识的边界
根据当前经验库，经验规则的核心边界应始终保持为：

- 经验只能影响合法动作集合内的排序
- 经验不能替代规则真值
- 读人与反推只作为弱信号
- 不能把经验推断当成确定事实 :contentReference[oaicite:2]{index=2}

---

## 5. 使用约束

### 5.1 RAG 可以做什么
RAG 可以用于：

- 规则理解支持
- 经验检索支持
- 决策参考输出
- 调试解释
- 在接入 DeepSeek API 后，为模型提供受限上下文

### 5.2 RAG 不能做什么
RAG 不能用于：

- 直接替代规则引擎判断合法性
- 跳过合法动作生成器直接出牌
- 直接执行状态更新
- 推动 AI 使用当前未实现牌型
- 覆盖 `docs/SPEC.md` 与当前代码实现中的规则边界

### 5.3 DeepSeek API 的边界
即使接入 DeepSeek API，也必须满足：

- DeepSeek 只读取受限上下文
- DeepSeek 输出不直接成为动作真值
- DeepSeek 输出不能绕过 `legal_actions`
- DeepSeek 不可用时，本地规则主链路仍应正常运行

---

## 6. 检索要求

### 6.1 检索结果必须可追踪
每次检索结果应至少包含：

- 来源文档
- 所属层级
- 命中的正文片段
- 对当前决策提供的支持类型

### 6.2 所属层级要求
规则知识至少要能区分：

- 当前已实现规则
- 第一阶段待补规则
- 背景规则

经验知识至少要能区分：

- 当前已接入经验
- 当前已收集但未接入经验

### 6.3 冲突知识处理
如果检索结果与当前局面规则约束冲突，则该结果只能作为：

- 被拒绝的参考信息
- 调试日志中的 rejected 证据
- 元数据记录

不能进入：

- 规则真值链路
- 动作合法性判断
- 状态更新裁决

---

## 7. 规则与经验的优先级关系

第一阶段中，优先级必须固定为：

1. **规则引擎为唯一真值**
2. **合法动作生成器决定 AI 可选动作边界**
3. **RAG 规则知识用于减少误解**
4. **RAG 经验知识只用于合法动作集合内排序与解释**
5. **经验建议不得改变动作合法性判断**
6. **经验建议不得覆盖规则引擎拒绝结论**
7. **未实现规则不得通过知识库正文被提前激活**
8. **DeepSeek API 接入后，模型输出仍受以上约束**

这和当前经验库中“经验只在合法动作集合内排序，不改变合法性判断”的表述保持一致 :contentReference[oaicite:3]{index=3}

---

## 8. 与代码实现的一致性要求

### 8.1 一致性原则
在第一阶段，规则知识库与经验知识库必须与当前代码实现保持同步。

这意味着：

- 当前已实现规则必须在 `guandan_rules.md` 中有一致表述
- 当前未实现但计划补齐的规则必须显式标注为“待补”
- 背景规则不得被当作当前可执行规则
- 当前已接入的经验必须与基础 AI 当前启发式一致
- 当前未接入的经验不得在代码中被假定为已生效

### 8.2 更新顺序要求
若后续继续补规则，建议顺序为：

1. 先改 `SPEC.md`
2. 再改 `guandan_rules.md`
3. 再改 `basic_human_experience.md`
4. 再补代码实现
5. 再补测试
6. 最后再做 RAG/DeepSeek 联调

---

## 9. DeepSeek API 接入要求

在第一阶段接入 DeepSeek API 后，RAG 链路必须满足：

1. DeepSeek 只读取经过约束的规则/经验上下文
2. DeepSeek 输出不直接成为动作真值
3. DeepSeek 输出不能绕过 `legal_actions`
4. DeepSeek 不可用时，系统仍应保持本地规则主链路可运行
5. DeepSeek 的作用应是：
   - 规则解释
   - 经验参考
   - 辅助说明
   而不是替代引擎决策

---
