# RAG_KB

## 1. 目标

当前阶段中的 RAG 只服务于两个核心目标：

1. 限定 AI 对当前单局掼蛋核心规则的理解范围
2. 引入基础人类经验，使 AI 在合法动作集合内更像正常人出牌

在后续接入 DeepSeek API 后，RAG 还承担第三个目标：

3. 为模型提供受限的规则与经验上下文，但不改变规则真值链路

---

## 2. 当前阶段的主线边界

当前阶段的唯一规则主线是：

> 单局掼蛋核心规则闭环。

因此 RAG 只能围绕以下内容提供支持：

- 单局规则理解
- 合法动作语义解释
- 牌型与压制关系说明
- 接风 / 名次 / 终局 / 平局判定说明
- 人类经验型排序建议

RAG 当前不应主动扩展到：

- 多局升级赛完整规则
- 打到 A 的长期比赛条件
- 复杂贡还 / 抗贡比赛制
- 完整地方规则百科
- 其他纸牌游戏规则

---

## 3. 主知识源约定

### 3.1 规则知识库主文件

当前阶段的**唯一主规则正文**为：

- `rag/rule_corpus/guandan_rules.md`

该文件是以下内容的主文本依据：

- 当前已实现规则的文字定义
- 当前阶段待补规则的文字定义
- 背景规则与完整规则说明

### 3.2 经验知识库主文件

当前阶段的**唯一主人类经验正文**为：

- `rag/experience_corpus/basic_human_experience.md`

该文件是以下内容的主文本依据：

- 当前基础 AI 已采用的经验规则
- 可用于解释当前 AI 选择逻辑的经验条目
- 当前已收集但尚未接入排序逻辑的经验条目

### 3.3 文档与代码的从属关系

当前阶段应遵循以下关系：

- `docs/SPEC.md`：规则目标规格与边界
- `docs/TESTS.md`：测试覆盖与验收口径
- `rag/rule_corpus/guandan_rules.md`：规则正文主文件
- `rag/experience_corpus/basic_human_experience.md`：经验正文主文件
- 代码实现：规则真值执行层
- RAG：规则 / 经验检索层
- DeepSeek API：后续受限模型支持层

也就是说：

- 规则正文不应再由多个文档平级重复定义
- 代码、测试、RAG 必须围绕同一套规则边界收敛
- RAG 永远不是规则真值源

### 3.4 场景化标签

知识库条目应尽量按 `适用场景` 标签组织，便于 RAG 在不同局面下做更精准检索。

常见标签示例：

- `通用`
- `my_info`
- `current_round`
- `other_players`
- `history`
- `legal_actions`

要求：

- 每条规则或经验尽量明确其适用场景
- 场景标签只用于检索和解释，不改变规则真值
- 同一条知识不应在多个正文位置重复堆叠描述，优先通过标签和短句归类

## 4. 知识库分层要求

## 4.1 规则知识库分层

`guandan_rules.md` 应显式区分以下三层内容：

### A. 当前已实现规则

指当前代码已经支持、当前测试已经验证、当前可以进入执行链路的规则。

这部分内容可以被：

- 规则检索
- 调试解释
- DeepSeek 上下文引用

直接用于当前局面的规则支持。

### B. 当前阶段待补规则

指已经确定要纳入当前阶段，但当前代码尚未完成实现、或测试尚未完全覆盖的规则，例如：

- 尚未完全补齐的 DeepSeek 决策稳定性验证
- 尚未完全补齐的 AI / RAG / CLI 契约回归

这部分内容可以写入知识库正文，但必须显式标注：

- 当前未接入执行真值链
- 当前不能作为合法性判断依据
- 当前不能直接驱动动作生成

### C. 背景规则或完整规则说明

指帮助理解完整掼蛋的背景规则、扩展规则或未来可能用到的规则。

这部分只能用于：

- 背景说明
- 人工阅读
- 未来扩展参考

不得被误用为当前可执行规则。

---

## 4.2 经验知识库分层

`basic_human_experience.md` 应显式区分以下两层：

### A. 当前已接入 AI 的经验

这部分经验应与当前基础 AI 的启发式保持一致，例如：

- 记录高价值信息
- 尽量减少未来出牌轮次
- 不无意义提前消耗炸弹
- 不无意义拆解基础组合
- 队友少牌时优先配合
- 读人与反推只作为弱信号，不替代规则真值

这些经验可直接用于：

- 合法动作集合内排序
- 决策依据输出
- 调试解释
- DeepSeek 上下文支持

### B. 当前已收集但尚未接入排序逻辑的经验

这部分经验可以存在于知识库中，但必须明确其状态：

- 当前尚未进入执行排序
- 当前只是候选经验
- 当前不应在代码实现中被假定为已生效

### C. 记牌相关经验

知识库可以加入基于出牌历史推断剩余牌的弱信号经验，例如：

- 某点数已大量露出时，对应炸弹密度下降
- 关键控制牌已被频繁打出时，对手剩余控制力可能下降
- 某玩家长期未出某类牌时，只能作为弱信号参考

要求：

- 必须标注为“弱信号”或“未接入执行排序”
- 只能作为解释与排序候选，不能当作确定事实
- 不能覆盖或篡改规则正文中的合法性描述

---

## 5. 来源约束

### 5.1 允许的知识来源

当前阶段默认只允许从以下两份文件中读取正文知识：

- `rag/rule_corpus/guandan_rules.md`
- `rag/experience_corpus/basic_human_experience.md`

如无额外文档更新，不应再加入第三份具有同等地位的规则正文或经验正文。

### 5.2 内容质量要求

规则知识与经验知识必须满足：

- 与当前项目采用的规则口径一致
- 与 `docs/SPEC.md` 中声明的当前边界一致
- 不包含其他纸牌游戏规则
- 经验条目不与规则冲突
- 不引入模糊、不确定、互相矛盾的知识

### 5.3 内容精简要求

知识库内容应保持精简、可检索、可维护：

- 优先写短句和可复用条目
- 避免把同一规则在多个位置长篇重复
- 对新增场景，优先补标签与短说明，而不是堆叠冗长背景
- 当条目过长时，应拆分为“规则摘要 + 场景标签 + 弱信号说明”三部分

### 5.4 当前经验知识边界

经验规则的核心边界必须始终保持为：

- 经验只能影响合法动作集合内的排序
- 经验不能替代规则真值
- 读人与反推只作为弱信号
- 不能把经验推断当成确定事实

---

## 6. 使用约束

## 6.1 RAG 可以做什么

RAG 可以用于：

- 规则理解支持
- 经验检索支持
- 决策参考输出
- 调试解释
- 在接入 DeepSeek API 后，为模型提供受限上下文

## 6.2 RAG 不能做什么

RAG 不能用于：

- 直接替代规则引擎判断合法性
- 跳过合法动作生成器直接出牌
- 直接执行状态更新
- 推动 AI 使用当前未实现牌型
- 覆盖 `docs/SPEC.md` 与当前代码实现中的规则边界

## 6.3 DeepSeek API 的边界

即使接入 DeepSeek API，也必须满足：

- DeepSeek 只读取受限上下文
- DeepSeek 输出不直接成为动作真值
- DeepSeek 输出不能绕过 `legal_actions()`
- DeepSeek 不可用时，本地规则主链路仍应正常运行

---

## 7. 检索要求

### 7.1 检索结果必须可追踪

每次检索结果应至少包含：

- 来源文档
- 所属层级
- 命中的正文片段
- 对当前决策提供的支持类型

### 7.2 所属层级要求

规则知识至少要能区分：

- 当前已实现规则
- 当前阶段待补规则
- 背景规则

经验知识至少要能区分：

- 当前已接入经验
- 当前已收集但未接入经验

### 7.3 冲突知识处理

如果检索结果与当前局面规则约束冲突，则该结果只能作为：

- 被拒绝的参考信息
- 调试日志中的 rejected 证据
- 元数据记录

不能进入：

- 规则真值链路
- 动作合法性判断
- 状态更新裁决

---

## 8. 规则与经验的优先级关系

当前阶段中，优先级必须固定为：

1. **规则引擎为唯一真值**
2. **合法动作生成器决定 AI 可选动作边界**
3. **RAG 规则知识用于减少误解**
4. **RAG 经验知识只用于合法动作集合内排序与解释**
5. **经验建议不得改变动作合法性判断**
6. **经验建议不得覆盖规则引擎拒绝结论**
7. **未实现规则不得通过知识库正文被提前激活**
8. **DeepSeek API 接入后，模型输出仍受以上约束**

---

## 9. 与代码实现的一致性要求

### 9.1 一致性原则

在当前阶段，规则知识库与经验知识库必须与当前代码实现保持同步。

这意味着：

- 当前已实现规则必须在 `guandan_rules.md` 中有一致表述
- 当前待补规则必须显式标注为“待补”
- 背景规则不得被当作当前可执行规则
- 当前已接入经验必须与基础 AI 当前启发式一致
- 当前未接入经验不得在代码中被假定为已生效

### 9.2 更新顺序要求

若后续继续补规则，建议顺序为：

1. 先改 `docs/SPEC.md`
2. 再改 `docs/TESTS.md`
3. 再改 `rag/rule_corpus/guandan_rules.md`
4. 再改 `rag/experience_corpus/basic_human_experience.md`
5. 再补代码实现
6. 再补测试
7. 最后再做 RAG / DeepSeek 联调

---
