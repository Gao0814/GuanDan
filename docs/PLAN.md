# PLAN

## 1. 总体目标
在第一阶段完成一个可运行的掼蛋基础系统，支持：

- 掼蛋引擎闭环
- 4 AI 自动完整对局
- 开发者全信息调试
- 基础规则型、经验型 AI
- RAG 规则库与经验库接入
- DeepSeek API 接入
- 基础测试与日志
- 后续人类接入的基础接口预留

---

## 2. 当前阶段判断
当前项目已经完成：

- 项目骨架
- 当前最小规则子集下的牌型识别
- 当前最小规则子集下的合法动作生成
- 当前最小规则子集下的规则校验与状态转移
- 4 AI 自动对局基础闭环
- 基础调试输出
- 基础经验型 AI
- 本地 RAG 最小链路

当前尚未完成：

- 第一阶段完整基础规则补全
- 大王、小王处理
- 关键基础牌型补全
- 规则库与经验库正文补全
- DeepSeek API 接入
- 在补全规则下重新完成稳定性回归

---

## 3. 第一阶段剩余实施原则
后续工作必须遵守以下原则：

1. 先补规则，再接 API
2. 先补知识库，再接模型支持
3. 先保证完整基础规则稳定，再考虑第二阶段
4. 任何新增牌型都必须同步补：
   - 牌型识别
   - 合法动作生成
   - 压牌规则
   - 状态转移
   - 测试
   - 知识库正文

---

## 4. 第一阶段剩余实施步骤

### Step A：冻结第一阶段最终规则范围
目标：
- 明确第一阶段最终必须支持的基础牌型与规则
- 明确大王、小王是否及如何纳入
- 明确哪些规则仍不属于第一阶段

完成标准：
- `SPEC.md`
- `TESTS.md`
- `RAG_KB.md`
- 规则库正文

四者口径一致

---

### Step B：补全基础掼蛋牌型与规则
目标：
- 补全第一阶段必须支持的核心牌型
- 补全大王、小王相关规则
- 补全对应动作生成、压牌规则、状态转移

完成标准：
- 新牌型在识别、生成、比较、状态更新链路上全部打通
- 不破坏当前已实现规则子集

---

### Step C：补全知识库正文
目标：
- 补全 `guandan_rules.md`
- 补全 `basic_human_experience.md`
- 区分当前已实现、待补、背景规则

完成标准：
- 知识库与当前实现保持一致
- 未实现规则不会被误用为当前可执行规则

---

### Step D：接入 DeepSeek API
目标：
- 通过 `.env` / `config.py` 接入 DeepSeek API
- 在允许范围内为 RAG/AI 提供模型支持
- 不改变规则真值链路

完成标准：
- API 可调用
- 调用失败不破坏本地规则链路
- 模型输出不替代引擎裁决

---

### Step E：完整回归与第一阶段收口
目标：
- 在更完整规则范围下重跑 4 AI 自动对局
- 完成第一阶段回归测试
- 验证调试输出、知识库一致性、API 接入边界

完成标准：
- 4 AI 自动对局稳定
- 非法动作率为 0
- 规则库与经验库与实现一致
- 可进入第二阶段

---

## 5. 当前阶段不允许实现的内容
- 强化学习
- 自主学习
- MCTS
- 蒙特卡洛搜索优化
- 自博弈训练
- 第二阶段与第三阶段内容

---

## 6. 当前风险点
1. 第一阶段规则范围若继续不冻结，会导致反复返工
2. 知识库正文若超前于实现，会导致 RAG 与代码口径不一致
3. API 若过早接入，会增加调试复杂度
4. 若在规则未补全前进入第二阶段，学习环境会不稳定

---

## 7. 当前建议顺序
建议严格按以下顺序推进：

1. 改文档
2. 补规则
3. 补知识库
4. 接 API
5. 做完整回归
6. 再进入第二阶段
